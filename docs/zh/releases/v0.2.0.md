# v0.2.0

DataFoundry 0.2 将首个受控 Data Agent 工作台，扩展为支持并发分析、证据追问、资产复用和正式 Web 部署的有状态工作流。

## 重点更新

### Data Link：面向 Data Agent 的自研语义基础

我们最近开源的数据知识图谱 [Data Link](https://github.com/datagallery-lab/datalink)，在 0.2 中正式进入 DataFoundry 生态，成为同源自研的语义基础。Schema 只告诉 Agent 「有哪些字段」；Data Link 进一步告诉它「这些字段意味着什么、数据如何连接、哪些关系值得信任」。

- 构建双层图谱：表和字段等结构节点始终锨定真实数据，共享的 Concept 和 Entity 节点统一组织业务含义。
- 将外键、血缘等显式关系，与 `joinable`、`semantic_synonym`、`correlated`、`distribution_similar` 等推断关系统一入图，并保留置信度。
- 为 Agent 提供面向问题的检索界面，不需要把整个数据目录塞进 prompt。
- 通过 MCP 和 REST 对外提供能力。DataFoundry 可发现已配置的 Data Link/DataGraph 服务，并在 Web 工作台中提供图谱查看、自然语言探索、表管理和重建工作流。
- 为复杂分析提供更强的语义起点：减少盲目 JOIN、减少字段猜测，让 Agent 从物理 Schema 更稳定地走向业务概念。

Data Link 的目标是统一组织表格、文档、幻灯片和图像。DataFoundry 0.2 已接入其图谱与 Agent 检索界面；Data Link 服务仍独立部署和配置，两个开源项目可以各自演进。

### 有状态分析与会话控制

- 多个数据任务会话可同时运行，各自的 live state 不互相污染。
- run 进行中可排队、编辑、立即发送或删除后续问题。
- 刷新页面或切换会话后，可恢复已持久化的消息、工具调用、run events、产出和终态。
- 可从早期用户问题重新提问，或从持久化 checkpoint 创建分支。原路径保留，并可在多个分支间显式切换。

### 证据、Trace 与语义探索

- 下一个问题可引用完整产出，或选中的表格区域和文本片段。
- Artifact、SQL audit、trace step、schema、preview 和 knowledge 引用会先在服务端解析，再进入受控 run context。
- 可通过语义 Trace DAG 查看 run、step、tool call、checkpoint、output 及它们之间的关系。
- 配置兼容的 Data Link MCP 服务后，可在工作区图中探索表、字段、概念、实体和关系边。

### 产出与工作区资产

- 会话产出可独立于实时对话恢复。
- 可预览表格、图表、报告、SQL 和文件，并导出支持的表格和图表格式。
- 文件可先上传到当前会话，再将支持的文件或文件型产出提升为跨会话工作区资产。
- 后续问题可通过 mention 引用可复用资产，从已有材料继续分析。

### Web 部署与引导

- 内置密码认证，覆盖注册、验证、登录、密码重置、会话管理、CSRF 保护和安全 Cookie。
- 正式部署中，浏览器的 API 和 CopilotKit 流量经过同源 Next.js 代理。
- Web 工作台支持英文和简体中文。
- OpenAI-compatible 模型配置可先执行连接测试，再选用到 run。
- 首次引导与自动准备的 DTC Growth Review SQLite 案例降低首次体验成本。

### 运行时与终端工作流

- 在保持受控上下文编译与 prompt snapshot 顺序稳定的同时，兼容 provider 对消息内容的约束。
- 历史恢复和继续提问仍遵守 schema-first、只读 SQL 和审计边界。
- TUI 升级为 chat-first 交互，并补齐会话恢复、产出浏览、表格、输入历史、补全、滚动和响应式终端布局。

## 稳定性与 Bug Fixes

- **确定性模型上下文：**修复 prompt snapshot 顺序与 provider 消息兼容问题，让恢复历史、当前问题、assistant tool call 和 tool result 按正确顺序送达模型。
- **会话恢复：**消除会话切换竞态，从后端 checkpoint 恢复终态，保留 tool-call/result 配对，并防止旧会话回写覆盖当前对话。
- **会话交互：**修复消息滚动恢复、切换会话后的定位、输入框遮挡、提问队列状态、运行指示和停止行为。
- **SQL 与审计稳定性：**在传输前标准化 BigInt 等不可直接 JSON 序列化的 SQL 值，稳定 audit log ID，避免结果序列化失败导致 artifact 丢失。
- **数据源就绪状态：**收紧数据源选择、本地文件路径、连接状态和 schema-first 执行，让缺失必要数据上下文的 run 更早、更清楚地失败。
- **工具与 Trace 传递：**改善流式 tool result 标准化、失败展示、父子 step 关联、Token 用量、取消和刷新后产出恢复。
- **正式 Web 路径：**通过轻量路由入口减小首屏工作台 bundle，加固同源 API 代理，并完善后端启动与 readiness 报告。
- **发布校验：**修复 Web 工作台拆分后漂移的源码约束测试，并将 smoke tests 从已退役的 DuckDB demo 对齐到新 DTC 案例。

## 兼容性与已知边界

- 对现有 0.1 集成，本版本没有有意引入公开 REST 或 AG-UI 协议级 breaking change，但推荐部署和 demo 路径已调整。
- 已退役的 `api-duckdb-demo` 不再自动创建。`npm install` 现在会生成 DTC Growth Review SQLite fixture，API 再按 user/workspace 自动准备副本。
- Data Link 需要独立运行兼容的 Data Link/DataGraph MCP 与 REST 服务。
- 这是 pre-release。用于生产前，请按实际环境验证认证、Secret、数据源策略、备份、监控和反向代理。

## 升级说明

- 需要 Node.js 22 或更高版本。
- 拉取 0.2 后请重新执行 `npm install`，因为 workspace 依赖和 lockfile 已变更。
- 正式部署使用 `password` 认证与 `build` / `start`，必填认证变量见 [快速开始](../quick-start.md)。
- 修改 `NEXT_PUBLIC_*` 后需要重新构建 Web。
- Data Link 是集成界面，不是内置语义服务；需先配置兼容的 MCP Server。
- 生产部署仍需根据环境配置 Secret 管理、访问策略、审计导出、监控、备份和 TLS 终止。
