Mastra Memory 受控接入架构

metadata 仍是权威历史；Mastra Memory 先做派生投影，再按门禁进入 prompt。

实线：当前/Phase 1 点线：后续阶段 浅色卡片：单一职责模块

1. 权威历史层

API

AG-UI Run

客户端只需要发送当前 user message。`threadId` 是 session，`runId` 是本次执行。

DB

metadata.sqlite

`conversation_messages` 保存可复用自然语言轮次，`conversation_summaries` 保存覆盖范围摘要。

SVC

ConversationMemoryService

run claim 后读取 latest summary + 未摘要历史，生成唯一的权威入口 window。

不启用 Mastra MessageHistory 注入 raw history；否则会和这里的权威 window 重复。

2. Agent 执行层

CTX

Budgeted Messages

当前默认：`unsummarized recent messages + current user` 进入 DataAgent；compact summary 交给 Mastra Memory。

MAS

Mastra DataAgent

`@ag-ui/mastra` 传入 `memory={thread, resource}`，但 Memory 配置为受控模式。

GOV

MastraContextBudgetProcessor

每一步 ReAct prompt 仍经过 token budget、turn grouping、tool result 降级和 provider guard。

SUM

LLM Summarizer

`RUN_FINISHED` 后触发摘要生成；满足阈值才写 `conversation_summaries`。

3. Mastra Memory 派生层

M1

Memory Runtime

`lastMessages=false`，不让 Mastra 自动取 raw history。先复用 LibSQL storage 形态。

M2

WorkingMemory Shadow

`shadow` 模式只在 summary commit 后由后端 bridge 单向写入，不进入 prompt。

M3

Read-only Injection

当前默认模式。Mastra input processor 注入只读 WorkingMemory，入口消息不再带 tagged summary。

M4

Recall / OM Gate

Semantic Recall 与 Observational Memory 先 shadow 比较质量、延迟和审计性。

注入触发：非 resume 新 run claim 成功后，`createDataAgent` 前构造入口 messages。
摘要触发：`RUN_FINISHED` 后 flush assistant 文本；未摘要消息达到阈值才生成。
Memory 触发：summary commit 后 mirror；新 run 前 backfill；随后由 Mastra input processor 注入。
