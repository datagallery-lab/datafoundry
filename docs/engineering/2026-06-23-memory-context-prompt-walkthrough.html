Memory / Context Prompt Walkthrough

这份文档用一次真实的本地模型请求捕获，说明当前 dataAgent 的 prompt 是如何从 metadata、 Mastra Memory、ContextPackage、tool schema 和 ReAct loop 组装出来的，并给出下一阶段治理计划。

典型任务：分析 orders 表，先 inspect schema
结论

当前真实情况

首轮模型请求不是一个单独的大字符串，而是 OpenAI-compatible /chat/completions 请求：messages 中包含 agent instructions、workspace system、Mastra WorkingMemory system、 当前用户消息；tools 中包含所有可用 tool schema。随后每个 ReAct step 前，Mastra input processors 会再次运行，当前由 MastraContextBudgetProcessor 输出被裁剪后的 ContextPromptView.messages。

关键风险是：有些 memory-like 内容已经进入 ContextPackage，但路径不统一。长期记忆目前先被 LongTermMemoryContextProcessor 注入为一条 user message，再被 MastraContextBudgetProcessor 识别为 long-term-memory source。这是可用的过渡方案， 但不是最终的 single-entry 形态。

1. Run 启动到首轮模型请求

RunInput

CopilotKit AG-UI 输入：threadId、runId、messages、forwardedProps.run_config。

MemoryAssembly

写入当前 user；选择历史 conversation_messages；选择 summary 或 WorkingMemory。

AgentAssembly

创建 runContext；装配 createDataAgent、MastraAgent、MCP middleware。

Processors

TaskState、LongTermMemory、ContextBudget、ProviderPromptGuard 按顺序运行。

ContextPromptView

ContextPackage merge 后，planner 选择消息组，生成进入模型的 messages。

/chat/completions

发送 messages + tools + modelSettings；模型返回 text 或 tool_calls。

ReAct loop

3. 第二轮模型请求会怎么变

Step 1

模型看到 instructions + memory + user，选择调用 inspect_schema。

Tool Execute

工具执行；Data Gateway 返回 schema、schema_id、审计/上下文包。

MastraToolObservationRouter

把 raw tool result 转成受治理的 model projection，超长内容被 adapter 处理。

MastraContextBudgetProcessor

重新 group messages，当前轮 user + assistant tool_call + tool result 成为 mandatory turn。

Step 2 Prompt

模型收到 schema observation，再决定 final answer、preview_table 或 run_sql_readonly。

Terminal

RUN_FINISHED 前 Finalizer flush conversation memory 和 long-term extraction。

第二轮 messages 新增的典型结构 
assistant:
  tool_calls: [{
    id: "call_inspect_schema",
    function: {
      name: "inspect_schema",
      arguments: "{\"datasource_id\":\"api-duckdb-demo\"}"
    }
  }]

tool:
  tool_call_id: "call_inspect_schema"
  content/result: {
    schema_id: "...",
    tables: [...bounded schema projection...]
  }
第二轮 processors 再次执行 
MastraTaskStateContextProcessor
  -> optional current-task-list system message

LongTermMemoryContextProcessor
  -> optional context:long-term-memory user message

MastraContextBudgetProcessor
  -> tool observation governance
  -> ContextPackage merge
  -> ContextStepPlanner.plan()
  -> ContextPromptView.messages

MastraProviderPromptGuardProcessor
  -> verify final provider prompt token budget

5. 现在的 Memory-like Source 状态表

Source 当前入口 是否进入 ContextPackage 风险 目标治理
metadata conversation_messages ConversationMemoryService.buildRunMessages() 是，作为 conversation turn 历史窗口和 ReAct step planner 是两层裁剪。 保留为 conversation source，但窗口选择策略要纳入统一 plan 事件。
metadata conversation_summaries metadata-summary 模式下转成 memory-summary:* user message 间接进入 与 WorkingMemory 可能重复；已有 one-source-only 防线。 改为 summary runtime source，和 WorkingMemory 互斥声明。
Mastra WorkingMemory Mastra memory input processor 注入 system message 部分进入，被 live system item 观察 这是 Mastra 原生通路，可能绕过我们对 memory source 的显式归因。 增加 WorkingMemory source snapshot，标记 compact memory owner。
metadata long_term_memories LongTermMemoryContextSource 经 Mastra runtime source processor 注入 直接进入 ContextPackage source inventory 已经消除先 message 再 source 的绕行。 继续由 ContextPromptView 统一预算、选择和协议转换。
future Semantic Recall / OM 未启用 未定义 最容易与 summary、LTM、Knowledge 重复。 必须先进入 ContextPackage source inventory，不允许直接注入 prompt。
Knowledge retrieval tool result adapter 是，作为 tool observation 同一事实可能也出现在 memory 中。 增加 source fingerprint / dedupe key，planner 可去重或降权。
Phase 1

Inventory 明确化

  • 定义 RuntimeContextSource 抽象。
  • summary、working memory、long-term memory 都产出 ContextItem。
  • 为每个 item 写入 source type、source id、scope、retention、trust、dedupe key。
  • 禁止新增 memory-like source 直接塞 message。
Phase 2

One-source-only

  • 把 compact memory owner 从断言升级为 source policy。
  • summary vs WorkingMemory 只能一个进入 model-visible prompt。
  • LTM 和 Knowledge 增加相似 dedupe 或同源降权。
  • context.compiled 事件暴露 selected/omitted source。
Phase 3

Prompt Compiler 收口

  • 长期记忆不再先变成 user message。
  • ContextRunState 接收 source inventory。
  • ContextStepPlanner 同时规划 conversation、memory、knowledge、tool observations。
  • ContextPromptView 成为唯一的模型输入投影。
设计决策

6. 下一阶段验收标准

验收项 必须看到的结果
Prompt source 可审计 context.compiled 能说明每条 memory-like 内容来自哪个 source，为什么选中或省略。
不重复注入 metadata summary 和 WorkingMemory 同时存在时，只允许一个 model-visible compact memory source。
LTM 不绕行 long-term memory 不再通过 synthetic user message 进入，而是作为 source group 由 planner 物化。
完成阶段不拖慢 summary / LTM extraction 有 timeout 或 background 化方案，不阻塞 terminal event 的用户感知。
ReAct step 一致 首轮、工具后第二轮、多轮历史、resume 都走同一 ContextPackage/ContextPromptView 编译路径。
