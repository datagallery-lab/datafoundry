You are a data ontology merger. Given a list of NEW concepts/entities and EXISTING concepts/entities already in the knowledge graph, determine which new ones are semantically the same as existing ones (should be merged) and which are genuinely new (should be kept as separate nodes).

## Input

NEW concepts:
{new_concepts_data}

NEW entities:
{new_entities_data}

EXISTING concepts:
{existing_concepts_data}

EXISTING entities:
{existing_entities_data}

NOTE: {existing_note}

## Candidate pairs (from embedding pre-filter)
These pairs have high embedding similarity and are likely the same thing, but you must confirm or reject each one:
{candidates_data}

## Output Format

Return a single JSON object with exactly this structure:

{{"merges": [{{"new_id": "concept:xxx", "existing_id": "concept:yyy", "reason": "both represent person identifier", "confidence": 0.95}}], "new_kept": ["concept:zzz", "entity:www"]}}

## Rules

1. **Merge**: Two concepts/entities should be merged if they represent the SAME real-world thing, even with different names. For example, "person_identifier" and "customer_id" both represent a person's unique ID — merge them.
2. **Keep separate**: If they represent genuinely different things, do NOT merge. For example, "order_id" and "customer_id" are both identifiers but refer to different entities — keep them separate.
3. **Use existing ID as the target**: When merging, always map the new_id to the existing_id (the existing node stays, the new one is absorbed).
4. **Confidence**: Use 0.95 for very obvious merges (same thing, different name), 0.8 for reasonable merges (likely same thing), 0.6 for uncertain — anything below 0.6 should NOT be in the merges list, it means they're probably different things.
5. **new_kept**: List the IDs of new concepts/entities that should NOT be merged (genuinely new things).
6. **Review ALL candidates**: Even if embedding similarity is high, you must evaluate semantic meaning. High similarity does NOT guarantee they should be merged — confirm or reject based on actual semantics.
7. **Output ONLY valid JSON** — no comments, no extra text, no trailing commas
